## 核心论点

最成功的 LLM agent 实现使用的是简单、可组合的模式，而不是复杂框架。先用最简单的方案，能测量到收益再增加系统复杂度。把"agentic systems"分成两类：

- **Workflow**：通过预定义代码路径编排 LLM 和工具。
- **Agent**：LLM 动态决定自身流程与工具使用。

## 基础构件：Augmented LLM

基础构件是 LLM + 增强能力：retrieval、tools、memory。现代模型能主动使用这些增强：自己生成检索查询、选择合适工具、决定保留哪些信息。落地重点：(1) 针对具体场景裁剪这些能力；(2) 提供简单、文档良好的接口。Model Context Protocol 是接入第三方工具的一种推荐方式。

## 五种 Workflow 模式

- **Prompt Chaining**：把任务拆成固定顺序的子步骤，每次 LLM 调用处理上一次输出；可在中间加程序化 gate/check 校验。适合可拆成固定子任务、用延迟换准确率的场景。例：先生成营销文案再翻译；先写大纲、校验大纲、再写全文。
- **Routing**：对输入分类后分流到专门处理。让不同类别各自优化、互不拖累。适合类别清晰、分开处理更有效的复杂任务。例：客服请求按"通用/退款/技术支持"分流；简单问题路由到小模型、复杂问题路由到强模型以平衡成本。
- **Parallelization**：并行多次 LLM 调用再聚合。两种变体：*Sectioning* 把独立子任务拆开并行（如 guardrail 与主回答并行、对多个评估维度并行打分）；*Voting* 同一任务多跑几次提升置信度/多视角（如并行做多次代码漏洞审查、多 prompt 评估内容合规）。适合子任务真正独立、或多视角能提升置信度时。
- **Orchestrator-Workers**：中心 LLM 动态拆分任务并委派给 worker LLM，再综合结果。与 parallelization 的区别在于子任务是按输入动态产生而非预先固定。适合步骤无法预测的复杂问题。例：影响文件数不确定的代码改动；多来源信息收集。
- **Evaluator-Optimizer**：一个 LLM 生成、另一个 LLM 给反馈，循环迭代。适合有明确评估标准、迭代能明显改善结果、且 LLM 能给出有意义反馈的场景。例：需要反复打磨的文学翻译；需要判断"是否还要继续检索"的多轮搜索分析。

## 自治 Agent

Agent 在迭代循环中工作：用户给指令或对话澄清任务，agent 自主规划并执行，必要时在检查点或遇阻时请求更多信息。关键：每一步都必须从环境获得"ground truth"反馈（工具结果、代码执行结果等）来评估进展。停止条件（如迭代上限）用于保持可控。实现通常很简单：LLM 基于环境反馈使用工具。成功高度依赖清晰、用心的工具设计。Agent 适合步骤不可预测、无法硬编码固定路径的开放式问题；在可信环境里能扩展任务，但成本更高、错误会累积，需要充分沙箱测试与护栏。例：跨多文件的软件工程任务；computer use。

## 三条核心实现原则

1. **Simplicity（简单）**：agent 设计保持简单。
2. **Transparency（透明）**：显式展示 agent 的规划步骤。
3. **ACI（Agent-Computer Interface）**：像做产品/API 文档一样认真打磨工具文档并充分测试工具规格。

## 工具工程（Prompt-engineering your tools）

工具格式显著影响 LLM 表现。建议：给模型足够 token 先"思考"再下笔；选择贴近训练语料自然文本的格式；去掉格式负担（避免让模型数行号、做字符串转义）；像做 prompt engineering 一样认真做 tool 设计——清晰描述 + 用法示例 + 边界情况、参数命名直观、用 workbench 充分测试、用 poka-yoke 防误用（如要求绝对路径而非相对路径）。作者提到在 SWE-bench 上花在工具上的优化时间比整体 prompt 还多。

## 实践应用附录

- **客服**：天然适合 agent——交互是对话式的、又需要访问外部数据和执行动作（查订单/知识库、退款、改工单）；成功可被用户侧明确定义，可做按成功结果计费。
- **编码 agent**：很有前景——解可被自动化测试验证，可用测试反馈迭代；问题结构清晰、产出质量客观可测。Claude 已能仅凭 PR 描述解决 SWE-bench Verified 中的真实 GitHub issue，但人工 review 对系统整体一致性仍不可或缺。

## 何时用什么

- 简单场景：优化单次 LLM 调用 + retrieval 即可。
- Workflow：任务定义清晰、需要可预测性。
- Agent：开放式、步骤数不可预测、需要自主决策且工具环境可信。
- 框架：只作为起点；要理解底层代码，不要被框架抽象掩盖。
